## I. Reflection Questions

1. What aspects of Rust's security model stood out to you? Was anything surprising or new compared to other lang you know?

2. Do you think existence of unsafe code in Rust undermines its sec promises? Why or Why not?

3. How does Rust compare to C or Java in terms of protections against vulnerabilities like memory corruption and injection attacks? 

4. What are some key limitations or weaknesses in Rust's security model that developers should be aware of? 

5. Should Rust be considered a "safe" lang? Why or why not? What tradeoffs did the designers have to make?

## II. Discussion Prompts:

1. How should non-Rust developers evaluate claims about Rust's security? What questions should they ask? 

2. How can Rust provide memory safety without garbage collection like Java? What is the borrow checker and how does it work?

3. What kinds of vulnerabilities, like SQL injection or OS command injection can still occur in Rust code? How should developers mitigate these?

4. What best practices should Rust developers follow to maxmimize security, even with Rust's Protection? 

5. Do you think Rust will continue gaining popularity? Should more critical software be written it? What are the pros and cons?
